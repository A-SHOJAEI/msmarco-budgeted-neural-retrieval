# msmarco-budgeted-neural-retrieval

This repo implements a **budgeted retrieval** pipeline for MS MARCO Passage Ranking: compare a BM25 baseline vs a dense bi-encoder, and (for dense) measure the **effectiveness vs budget tradeoff** when swapping FAISS index types (exact `FlatIP` vs compressed/approximate `IVF-PQ`). “Budget” is operationalized as:

- retrieval latency percentiles (`p50_ms`, `p95_ms`, `qps`)
- (dense only) index footprint in bytes (`index_bytes`)

The default run is a fast, offline **smoke** pipeline that validates end-to-end wiring.

## Problem Statement

Given a query, retrieve the top-*k* relevant passages under a fixed serving budget (latency + memory). The repo evaluates ranking quality with `nDCG@k`, `MRR@k`, and `Recall@k` computed from MS MARCO-style qrels (`rel > 0` treated as relevant). See `src/msmarco_bnr/eval/metrics.py`.

## Datasets And Provenance

### Smoke dataset (used by the committed artifacts)

`configs/smoke.yaml` uses `dataset.type: smoke_synthetic`, generated by `src/msmarco_bnr/data/smoke.py` into `data/smoke/` with MS MARCO-like filenames:

- `data/smoke/collection.tsv`
- `data/smoke/queries.dev.small.tsv`
- `data/smoke/qrels.dev.small.tsv`
- `data/smoke/triples.train.small.tsv`

This dataset is **synthetic** (not derived from MS MARCO). Relevance is constructed so query *qid* has a single relevant passage whose text is injected with a `topic{qid}` token.

### MS MARCO Passage Ranking (supported by code)

`dataset.type: msmarco_passage` downloads official MS MARCO artifacts (as wired in `src/msmarco_bnr/data/msmarco.py`) into `data/raw/msmarco/` and prepares a working tree in `data/msmarco/`:

```text
https://msmarco.blob.core.windows.net/msmarcoranking/collection.tar.gz
https://msmarco.blob.core.windows.net/msmarcoranking/queries.train.tsv
https://msmarco.blob.core.windows.net/msmarcoranking/queries.dev.small.tsv
https://msmarco.blob.core.windows.net/msmarcoranking/qrels.train.tsv
https://msmarco.blob.core.windows.net/msmarcoranking/qrels.dev.small.tsv
https://msmarco.blob.core.windows.net/msmarcoranking/triples.train.small.tsv
https://msmarco.blob.core.windows.net/msmarcoranking/triples.train.tsv
```

Only `collection.tsv` is extracted from `collection.tar.gz` (path-traversal safe extraction is enforced). The downloader keeps a local checksum manifest at `data/raw/msmarco/checksums.json` after first download.

## Methodology (What’s Implemented)

### Baseline: BM25

Implemented in `src/msmarco_bnr/bm25/`:

- `rank_bm25` backend (`BM25Okapi`) for small/smoke runs.
- optional Lucene/pyserini backend via the official CLI + `LuceneSearcher` (`k1=0.9`, `b=0.4`) in `src/msmarco_bnr/bm25/pyserini.py`.

### Dense retrieval: bi-encoder + FAISS

Implemented in `src/msmarco_bnr/models/biencoder.py`, `src/msmarco_bnr/training/train.py`, `src/msmarco_bnr/index/faiss_index.py`, and orchestrated by `src/msmarco_bnr/eval/pipeline.py`.

- Encoder: Hugging Face `AutoModel` with mean pooling over tokens.
- Projection: optional linear layer to `model.projection_dim`; embeddings are L2-normalized, so inner product is cosine similarity.
- Training objective: in-batch contrastive cross-entropy over `2B` candidates per batch (each query competes against all positives/negatives in the batch).
- Distillation (optional): cross-encoder teacher (`cross-encoder/ms-marco-MiniLM-L-6-v2` in `configs/msmarco_*.yaml`) provides soft scores over the aligned `(pos, neg)` pair per query; student matches via temperature-scaled KL.
- Indexing/search:
  - `FlatIP`: exact inner product search.
  - `IVF-PQ`: compressed approximate search with configurable `nlist`, `m`, `nbits`, `nprobe`.
  - Indexes are saved under `artifacts/faiss/<run_id>/*.faiss`; size is recorded as `index_bytes`.

### Evaluation and “budget”

`scripts/eval.py` writes `artifacts/results.json` via `src/msmarco_bnr/eval/pipeline.py`:

- Effectiveness: `nDCG@{10,100}`, `MRR@{10,100}`, `Recall@{10,100}` (configurable `eval.k_values`).
- Latency: per-query latency percentiles computed over `eval.latency_max_queries` (single-query searches, batch size 1) plus aggregate QPS.

`scripts/report.py` turns `artifacts/results.json` into the single table in `artifacts/report.md` (rendered by `src/msmarco_bnr/utils/report.py`).

## Baselines / Ablations (As Named In Code)

Runs emitted by `src/msmarco_bnr/eval/pipeline.py`:

- `bm25` (baseline): BM25 via `bm25.backend` (`rank_bm25` or `pyserini`).
- `dense_distill:{flatip,ivfpq}` (model): bi-encoder trained with distillation (only if `distillation.enabled: true` and training produced a checkpoint).
- `dense_nodistill:{flatip,ivfpq}` (ablation): same bi-encoder trained without distillation.

## Reproducibility / How To Re-Run

### Smoke run (matches the committed artifacts)

```bash
make all
cat artifacts/report.md
cat artifacts/results.json
```

### MS MARCO dev.small (BM25 + dense + FAISS)

Prereqs:

- Install heavy ML deps: `make setup-ml` (PyTorch + transformers + faiss-cpu).
- For Lucene BM25 baseline: install Java + `pyserini` (`.venv/bin/pip install -r requirements-pyserini.txt`).

Commands:

```bash
make setup-ml
.venv/bin/pip install -r requirements-pyserini.txt
make data   CONFIG=configs/msmarco_devsmall.yaml
make train  CONFIG=configs/msmarco_devsmall.yaml
make eval   CONFIG=configs/msmarco_devsmall.yaml
make report CONFIG=configs/msmarco_devsmall.yaml
```

Determinism controls are in `src/msmarco_bnr/utils/repro.py` and are toggled by `run.deterministic` in the YAML configs.

## Limitations (Current Implementation)

- The committed numbers are from a **synthetic smoke dataset**; they are not meaningful MS MARCO benchmarks.
- Dense eval encodes **all passages in-process** and `src/msmarco_bnr/eval/pipeline.py` loads `collection.tsv` into RAM; this is not viable for the full MS MARCO corpus without sharding/streaming.
- Distillation is intentionally cheap: it matches the teacher only on the aligned `(pos, neg)` pair per query (no full `B x 2B` teacher scoring).
- Latency measurement uses single-query searches in Python; it is useful for relative comparisons inside this repo but not a production-grade benchmark harness.
- IVF-PQ training samples are taken from the head of the embedding array (not randomized), which can bias quantizer training.

## Next Research Steps

1. Add a scalable encoder/indexer path: shard passage encoding and build FAISS indexes without holding the full corpus in RAM.
2. Implement hard-negative mining (BM25 mined negatives; optionally cross-encoder re-ranking for mining) and compare vs the current single-negative triples.
3. Extend distillation to richer candidate sets (in-batch candidates or mined negatives) while controlling teacher cost.
4. Run and version real MS MARCO results (dev.small and full dev), including FAISS `FlatIP` vs `IVF-PQ` tradeoffs with tuned `nlist/m/nbits/nprobe`.
5. Add additional budget knobs beyond index compression (INT8/FP16 embedding quantization, HNSW, batching, GPU FAISS) and report pareto fronts.
